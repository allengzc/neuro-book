# 爽文

用于把任意题材的剧情调整成更有商业网文阅读驱动力的节奏。这个 skill 是剧情设计基础上的补充，只提供节奏和读者体验框架，不替代“剧情规划流程”的多视角剧情设计、题材设定、世界书、角色设定或剧情系统。

使用本 skill 时，应同时具备剧情设计指导：先知道当前剧情的范围、角色处境、冲突功能和用户想要的方向，再用爽文模板强化危机、低谷、释放、奖励和钩子。

## 核心原则

- 主角必须有可感知的能动性，不能只被命运推着走。
- 压力要具体，读者要知道主角为什么危险、为什么难、为什么必须现在行动。
- 爽点不是单纯赢，而是“被压住的期待得到清晰释放”。
- 奖励要对应前面的压力、代价或选择，不能像随机掉落。
- 每次释放后要留下新问题、新目标或新威胁，让读者继续翻页。

## 前置模板

设计开局、章节段落或关键场景前，先套一遍这个节奏模板。可以按题材自由变形，不要求每一项都独立成章。

1. 危机：主角立刻遇到会改变处境的麻烦。
2. 信息：读者理解规则、利害关系和主角的短板。
3. 反抗：主角用性格、知识、资源或临场判断主动破局。
4. 升级：原方案不够用，敌人、环境或误解进一步加压。
5. 低谷：让主角短暂接近失败，读者明确感到“这次不好过”。
6. 释放：通过已铺垫的能力、选择、援手或规则完成反转。
7. 奖励：给出战利品、身份提升、关系变化、信息突破或情绪满足。
8. 钩子：把胜利后的新麻烦、新地图、新目标或更大真相抛出来。

## 后置校验

剧情草案完成后，用下面问题检查爽点是否成立：

- 主角是否主动做了至少一个改变局势的选择？
- 压力是否足够明确，而不是只有“很危险”的形容？
- 低谷是否来自前面铺垫的规则或敌我差距，而不是临时强行降智？
- 释放是否能追溯到前文铺垫，而不是无根反转？
- 奖励是否让读者感到“这段值得看”？
- 章末或段落末是否还有继续阅读的理由？

## 与剧情 Skill 协作

- 开局设计时，先用“剧情规划流程”或“开局剧情设计”判断范围、冲突和角色行动，再用本 skill 套节奏模板。
- 中长期剧情规划时，先用“剧情规划流程”完成多视角分析和用户确认，再用本 skill 做爽点校验。
- 如果 Plot 支持类型标记，低谷节点优先用 `despair`，释放节点优先用 `relief`，奖励兑现仍可用 `reward`。
- 不要把本 skill 的模板当成固定题材。玄幻、都市、科幻、悬疑、恋爱、群像都可以使用同一套压力与释放逻辑。
